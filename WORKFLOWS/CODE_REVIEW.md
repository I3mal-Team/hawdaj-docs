# CODE_REVIEW — تدفّق المراجعة

> **Trigger:** `review` / `راجع` / `code review` (diff / فرع / ملف / PR). المحور: `MASTER_WORKFLOW.md`.
> **مبدأ:** المراجعة **قراءة فقط** — لا تعديل بلا موافقة. أخرِج نتائج مصنّفة بالخطورة.

```
DIFF / BRANCH / FILE
 ↓
Read AI Docs (الميزة + STANDARDS)
 ↓
Review Dimensions (صحّة · أمان · أداء · معايير · تعريب)
 ↓
Findings (مصنّفة + موقع + إصلاح مقترح)
 ↓
(اختياري) Approval → Apply Fixes
 ↓
Update AI Docs (لو ظهر دَين/قرار)
```

## 1. المدخل
- diff الحالي / فرع / ملف / PR. حدّد الميزة من `FEATURE_INDEX.md` + اقرأ مستندها + `STANDARDS/`.

## 2. أبعاد المراجعة
- **صحّة (Correctness):** bugs، حالات حافّة، null/errors، منطق خاطئ.
- **أمان:** auth/authz، تسريب PII، injection، أسرار مكتوبة، عقود عامة. (راجع `DECISIONS/TECH_DEBT.md` — لا تكرّر تحذيرات معروفة إلا لو المكان تغيّر.)
- **أداء:** N+1، استعلامات غير مفهرسة، إعادة بناء زائدة، حجب متزامن.
- **معايير:** `STANDARDS/` (حالة Cubit، `Either`, تسمية، بنية، معالجة أخطاء).
- **تعريب/UX:** نصوص عبر `.tr()`، RTL، لا نصوص مكتوبة يدويًا.
- **اتساق منصّات:** لو عقد API تغيّر، الموبايل+الويب متوافقان؟

## 3. Findings
- سطر واحد لكل نتيجة: `path:line — [severity] المشكلة. الإصلاح.`
- خطورة: 🔴 High · 🟡 Medium · 🟢 Low. لا مديح، لا nitpicks شكلية إلا لو تغيّر المعنى.
- رتّب: High أولًا.

## 4. (اختياري) Apply Fixes
- لو المستخدم قال `--fix` أو وافق: طبّق النتائج المتّفق عليها. غير كده → قراءة فقط.

## 5. Update AI Docs
- نتيجة أمنية/معمارية جديدة مؤكّدة → `DECISIONS/TECH_DEBT.md`.
- قرار تصميم → `DECISIONS/ARCHITECTURE_DECISIONS.md`.
- لا تحديث لو مراجعة عابرة بلا تغيير.

> بديل جاهز: `/code-review` (أداة الهارنس) لمراجعة الفرع الحالي/PR.

## سياسات إلزامية (Governance)
- **المراجعة تقيّم أثر diff قائم** حسب أبعاد [[Impact_Analysis]] (لم يُنشئه المراجع، بل يتحقّق منه).
- **إن أدّت المراجعة إلى تطبيق إصلاحات (`--fix`):** استوفِ [[Completion_Gate]] — أي بند ناقص (ومنه مزامنة التوثيق) = ❌ INCOMPLETE. (مراجعة قراءة فقط بلا تعديل → لا تنطبق البوابة.)
