# Impact_Analysis — تحليل الأثر (إلزامي قبل أي تنفيذ)

> **سياسة حاكمة.** ممنوع بدء أي تنفيذ (كود) قبل إتمام تحليل الأثر وعرضه ونيل الموافقة (بوابة §2.7 في `CLAUDE.md`).
> يُرجَع لهذا المستند من كل تدفّق في `WORKFLOWS/` وفي `CLAUDE.md`. لا يُكرَّر محتواه — يُشار إليه.

---

## 1. لماذا (Why)
- النظام **3 طبقات تتشارك API واحدًا** (موبايل + ويب + باك). تغيير أحادي الجانب يكسر الآخر بصمت.
- تحليل الأثر يكشف **الآثار الجانبية، التبعيات، والانحدار** قبل كتابة سطر — أرخص من إصلاح بعد الكسر.
- يمنع scope creep ويجعل الخطة والموافقة مبنيّتين على وقائع لا افتراضات.

## 2. متى (When)
- **قبل كل تنفيذ** في كل تدفّق: BUG_FIX · NEW_FEATURE · REFACTOR · PERFORMANCE · SECURITY. (CODE_REVIEW يستخدمه لتقييم diff قائم.)
- يأتي **بعد تحميل السياق** (قائمة `CLAUDE.md §2`) و**قبل الخطة والموافقة**.
- لا استثناء إلا لو المستخدم قال صراحةً "نفّذ بدون تحليل".

## 3. كيف (How)
1. حمّل السياق: `FEATURE_INDEX` → مستند الميزة → `FEATURE_GRAPH` (التبعيات) → `API_INDEX`/`DATABASE_INDEX` → `KNOWN_ISSUES`.
2. حدّد الميزة/الميزات المتأثّرة وتدفّق تنفيذها (UI→Cubit→Repo→API→Controller→Model→DB).
3. تتبّع الأثر عبر **كل بُعد** في القالب (§4). لا تتخطَّ بُعدًا — اكتب "لا أثر" صراحةً إن لم يوجد.
4. صنّف مستوى الخطر، وعدّد التبعيات/الكسر/الترحيل/التراجع/نطاق الاختبار.
5. اعرض النتيجة بالقالب، ثم انتقل للخطة (خيار/اثنان) ثم **بوابة الموافقة**.

## 5. مصفوفة مستوى الخطر
| المستوى | متى | الإجراء |
|---------|-----|---------|
| 🟢 Low | تغيير معزول، بلا عقد/DB، عميل واحد | مراجعة عادية + smoke |
| 🟡 Medium | يمسّ منطقًا مشتركًا أو أداءً أو UX | مراجعة + regression مركّز |
| 🟠 High | يمسّ عقد API/DB/auth أو عدّة ميزات | خطة دقيقة + regression موسّع + تنسيق العملاء |
| 🔴 Critical | أمني/تدمير بيانات/تعطّل إنتاج/تغيير كاسر واسع | تأكيد صريح + خطة تراجع + اختبار مكثّف + كتابة عادية (لا اختصار) |

## 6. القاعدة الذهبية (تُطبَّق دائمًا)
تغيير يمسّ **عقد API** → **Backend أولًا** → ثم الموبايل والويب على العقد الجديد → ثم مزامنة التوثيق. أي أثر عابر للريبوهات **يُوثَّق قبل التنفيذ**.
